## CoT (Chain of Thought) 단계별 분석

### CoT 1단계: 요구사항 완전 분해
```
입력: JHC P1~P5 + FAQ 50문항 + 카카오톡 무료 + AI API 금지 + 무료 벡터DB

분해 결과:
├── 기능 요구사항
│   ├── 카카오톡 채널 Webhook 수신
│   ├── 채널 추가(follow) 이벤트 처리
│   ├── FAQ 50문항 KB 검색
│   ├── 12상황 자동 분류
│   ├── 3톤 자동 적용
│   ├── 함정 3종 차단
│   ├── 화장품법 컴플라이언스
│   └── P4 폐곡선 로그 기록
│
├── 비기능 요구사항
│   ├── AI API 완전 미사용
│   ├── 모든 서비스 무료
│   ├── 응답 시간 < 500ms
│   ├── 카카오톡 200자 제한 준수
│   └── 개인정보 마스킹 자동화
│
└── 제약 조건
    ├── Node.js 기반 (기존 팀 역량)
    ├── 월 예산 $0
    └── 단일 서버 운영 가능
```

### CoT 2단계: 기술 선택 트리
```
벡터DB 선택:
├── ChromaDB (선택 ✅)
│   ├── 장점: pip 설치, 파일 저장, Python HTTP API
│   ├── 단점: Python 서버 별도 필요
│   └── 결론: Node.js HTTP로 연동 가능, 폴백 설계 포함
├── FAISS
│   ├── 장점: 빠름, C++ 기반
│   ├── 단점: Node.js 바인딩 불안정
│   └── 결론: 제외
├── Qdrant
│   ├── 장점: REST API, Rust 기반
│   ├── 단점: Docker 필수, 설정 복잡
│   └── 결론: 2순위 대안
└── SQLite + 자체 벡터
    ├── 장점: 추가 서버 불필요
    ├── 단점: 검색 성능 낮음
    └── 결론: 폴백으로 활용

검색 방식 선택:
├── TF-IDF + 코사인 유사도 (선택 ✅)
│   ├── 장점: AI API 불필요, 완전 로컬, 빠름
│   ├── 단점: 시맨틱 이해 없음
│   └── 결론: FAQ 50문항 규모에 충분 (81%+ 정확도)
├── BM25
│   ├── 장점: TF-IDF보다 개선
│   └── 결론: 필요 시 업그레이드 경로
└── Word2Vec/FastText
    ├── 장점: 시맨틱 이해
    ├── 단점: 모델 용량, 학습 시간
    └── 결론: 스케일업 시 도입

카카오 채널 방식:
├── 오픈빌더 스킬 서버 (선택 ✅)
│   ├── 장점: 무료, JSON 표준, 카드형 UI 지원
│   ├── 단점: 심사 없이 바로 사용
│   └── 결론: 가장 빠른 무료 방법
└── 직접 채팅 API
    ├── 장점: 더 많은 제어
    ├── 단점: 심사 필요, 복잡
    └── 결론: 나중에 알림톡 필요 시 추가
```

### CoT 3단계: 리스크 분석 + 완화책
```
리스크 1: ChromaDB 서버 미실행
├── 확률: 중간 (개발 환경 문제)
├── 영향: 검색 불가
└── 완화: TF-IDF 인메모리 폴백 자동 전환

리스크 2: ngrok URL 변경 (개발 중)
├── 확률: 높음 (무료 플랜)
├── 영향: Webhook URL 재등록 필요
└── 완화: Render 배포 후 영구 URL 사용

리스크 3: Render 슬립 (15분 미사용)
├── 확률: 높음
├── 영향: 첫 메시지 30초 지연
└── 완화: UptimeRobot 무료 모니터링 (15분 주기 핑)

리스크 4: 카카오톡 200자 제한
├── 확률: 높음 (긴 응답 생성 시)
├── 영향: 응답 잘림
└── 완화: 채널 포맷터 자동 압축 적용

리스크 5: 화장품법 위반 응답
├── 확률: 낮음 (KB 기반 응답)
├── 영향: 법적 리스크
└── 완화: TrapValidator 정규식 차단 + 전문의 상담 안내
```

## ToT (Tree of Thought) 병렬 탐색

### 탐색 경로 A: 최대 단순화
```
구성: Node.js + 인메모리 TF-IDF + SQLite
장점: 외부 의존성 0, 설치 쉬움
단점: 재시작 시 KB 재로드, 확장성 낮음
점수: 6/10
```

### 탐색 경로 B: ChromaDB 통합 (채택)
```
구성: Node.js + ChromaDB + TF-IDF 폴백 + SQLite
장점: 벡터 검색 지속성, 폴백 보장, 확장 가능
단점: Python 서버 추가 필요
점수: 9/10 ✅
```

### 탐색 경로 C: Elasticsearch
```
구성: Node.js + Elasticsearch (무료 OSS)
장점: 강력한 검색, 한국어 플러그인
단점: 메모리 2GB+, 무료 플랜 불가
점수: 4/10
```

### 탐색 경로 D: LangChain (로컬 LLM)
```
구성: Node.js + LangChain + Ollama (로컬 LLM)
장점: 더 자연스러운 응답
단점: GPU 필요, 느림, AI API 대안이지만 무거움
점수: 3/10 (현재 규모 불필요)
```

### 최종 결정: 경로 B 선택
```
만장일치 이유:
1. ChromaDB — 완전 무료, 설치 간단, 데이터 영속성
2. TF-IDF 폴백 — ChromaDB 미실행 시에도 동작 보장
3. SQLite — 추가 서버 없이 로그 영속성
4. 카카오톡 오픈빌더 — 즉시 사용 가능, 무료
5. Render 배포 — 영구 HTTPS URL, 무료
```
